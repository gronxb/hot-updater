## Installation

```package-install
npm install @hot-updater/plugin-core --save-dev
```

## Overview

Build a custom storage plugin when the built-in providers do not match your
storage backend.

The contract is intentionally small: teach Hot Updater how to save bundle
artifacts, delete them, materialize them for deploy-time work, read small
runtime metadata, and turn stored bundles into client-downloadable URLs. A
storage plugin should not know about database rows, rollout rules, or update
selection.

All storage plugins use `createStoragePlugin`. The same plugin shape can run in
CLI commands, console server actions, serverless functions, and edge runtimes
as long as the implementation only uses APIs available in that runtime. If a
platform needs bindings such as Cloudflare R2, pass those bindings directly in
the storage config before handing the plugin to `createHotUpdater`.

```typescript
import { createHotUpdater } from "@hot-updater/server";
import { r2Storage } from "@hot-updater/cloudflare/worker";

export const createHandler = (request: Request, env: Env) =>
  createHotUpdater({
    storages: [
      r2Storage({
        bucket: env.BUCKET,
        jwtSecret: env.JWT_SECRET,
        publicBaseUrl: new URL(request.url).origin,
      }),
    ],
  });
```

## Implementation Checklist

Before writing code, decide these values:

| Item                | Example                                      | Notes                                                                    |
| ------------------- | -------------------------------------------- | ------------------------------------------------------------------------ |
| `name`              | `customStorage`                              | Used in diagnostics. Keep it stable and human-readable.                  |
| `supportedProtocol` | `custom-s3`                                  | Must match every `storageUri` the plugin returns.                        |
| Storage URI shape   | `custom-s3://bucket/releases/bundle.zip`     | Include enough information to locate the object later.                   |
| Upload key policy   | `basePath / bundleId / filename`             | Use `createStorageKeyBuilder` unless the provider needs custom behavior. |
| Download URL policy | Signed URL, public CDN URL, or worker route  | `getDownloadUrl` must return HTTP(S).                                    |
| Text read policy    | SDK call, binding read, or authenticated API | `readText` is for small metadata such as `manifest.json`.                |

## Plugin Shape

Use `createStoragePlugin` and return only the operations your provider
supports. Hot Updater will throw a clear diagnostic if a workflow needs an
operation that is absent.

```typescript
import { createStoragePlugin } from "@hot-updater/plugin-core";

export const myStorage = createStoragePlugin<MyStorageConfig>({
  name: "myStorage",
  supportedProtocol: "custom",
  factory: (config) => ({
    async upload({ key, source }) {
      return { storageUri: `custom://bucket/${key}` };
    },
    async delete({ storageUri }) {
      // Delete the object referenced by storageUri.
    },
    async downloadFile({ storageUri, filePath }) {
      // Download the object referenced by storageUri into filePath.
    },
    async getDownloadUrl({ storageUri }) {
      return { fileUrl: "https://example.com/path/bundle.zip" };
    },
    async readText({ storageUri }) {
      return "{}";
    },
  }),
});
```

## Operation Contracts

Each method should validate that the incoming `storageUri` belongs to the
expected provider and bucket before touching storage. That guard prevents
accidentally deleting or reading from the wrong namespace when multiple storage
plugins are configured.

### `upload`

Uploads an artifact and returns the storage URI that Hot Updater saves in the
database.

```typescript
upload: (params: {
  readonly key: string;
  readonly source:
    | { readonly kind: "file"; readonly filePath: string }
    | {
        readonly kind: "bytes";
        readonly data: ArrayBuffer | Uint8Array | string;
        readonly contentType?: string;
      };
}) => Promise<{ readonly storageUri: string }>;
```

`key` is the logical upload path, usually the bundle ID or a child path under
the bundle ID. File-capable providers can use `getStorageUploadFilePath(source)`
to accept local files. Edge runtimes that cannot read local files should reject
`source.kind === "file"` and accept byte uploads only.

Return a provider URI, not a public URL:

```typescript
return {
  storageUri: `custom-s3://${bucketName}/${storageKey}`,
};
```

### `downloadFile`

Downloads a storage object into a local path.

```typescript
downloadFile: (params: {
  readonly storageUri: string;
  readonly filePath: string;
}) => Promise<void>;
```

Use this for provider-native downloads. This enables copy promotion and bundle
diffing for providers that cannot produce a `fetch()`-able URL from deploy-time
credentials.

### `delete`

Deletes the storage object referenced by `storageUri`.

```typescript
delete: (params: { readonly storageUri: string }) => Promise<void>;
```

Validate the bucket or namespace before deleting. If your provider supports
folder-like prefixes, keep the delete behavior explicit and predictable.

### `getDownloadUrl`

Returns an HTTP(S) URL that an app can download.

```typescript
getDownloadUrl: (params: {
  readonly storageUri: string;
}) => Promise<{ readonly fileUrl: string }>;
```

The returned URL must be usable by the client. Server code rejects local file
paths and non-HTTP(S) URLs.

### `readText`

Reads small control-plane artifacts directly from storage.

```typescript
readText: (params: { readonly storageUri: string }) => Promise<string | null>;
```

Use the provider SDK, runtime binding, or authenticated API. Do not generate a
public download URL and fetch that URL again. Hot Updater uses this path for
metadata such as `manifest.json`.

Return `null` when the object does not exist. Throw for permission errors,
invalid storage URIs, provider outages, and malformed provider responses.

## Complete Example

This example implements an S3-compatible storage plugin. The same shape works
for other object storage providers if you replace the SDK calls.

```typescript title="customStorage.ts"
import fs from "node:fs/promises";
import path from "node:path";

import {
  DeleteObjectCommand,
  GetObjectCommand,
  S3Client,
  type S3ClientConfig,
} from "@aws-sdk/client-s3";
import { Upload } from "@aws-sdk/lib-storage";
import { getSignedUrl } from "@aws-sdk/s3-request-presigner";
import {
  createStorageKeyBuilder,
  createStoragePlugin,
  getContentType,
  getStorageUploadFilePath,
  parseStorageUri,
} from "@hot-updater/plugin-core";

export interface CustomStorageConfig extends S3ClientConfig {
  readonly bucketName: string;
  readonly basePath?: string;
}

export const customStorage = createStoragePlugin<CustomStorageConfig>({
  name: "customStorage",
  supportedProtocol: "custom-s3",
  factory: (config) => {
    const { bucketName, basePath, ...clientConfig } = config;
    const client = new S3Client(clientConfig);
    const getStorageKey = createStorageKeyBuilder(basePath);

    const parseAndValidateStorageUri = (storageUri: string) => {
      const parsed = parseStorageUri(storageUri, "custom-s3");

      if (parsed.bucket !== bucketName) {
        throw new Error(
          `Bucket name mismatch: expected "${bucketName}", but found "${parsed.bucket}".`,
        );
      }

      return parsed;
    };

    return {
      async upload({ key, source }) {
        const filePath = getStorageUploadFilePath(source);
        const filename = path.basename(filePath);
        const storageKey = getStorageKey(key, filename);
        const upload = new Upload({
          client,
          params: {
            Bucket: bucketName,
            Key: storageKey,
            Body: await fs.readFile(filePath),
            ContentType: getContentType(filePath),
          },
        });

        await upload.done();

        return {
          storageUri: `custom-s3://${bucketName}/${storageKey}`,
        };
      },

      async delete({ storageUri }) {
        const { key } = parseAndValidateStorageUri(storageUri);

        await client.send(
          new DeleteObjectCommand({
            Bucket: bucketName,
            Key: key,
          }),
        );
      },

      async downloadFile({ storageUri, filePath }) {
        const { key } = parseAndValidateStorageUri(storageUri);
        const response = await client.send(
          new GetObjectCommand({
            Bucket: bucketName,
            Key: key,
          }),
        );
        const bytes = await response.Body?.transformToByteArray();

        if (!bytes) {
          throw new Error("Object body is empty");
        }

        await fs.mkdir(path.dirname(filePath), { recursive: true });
        await fs.writeFile(filePath, bytes);
      },

      async getDownloadUrl({ storageUri }) {
        const { key } = parseAndValidateStorageUri(storageUri);
        const command = new GetObjectCommand({
          Bucket: bucketName,
          Key: key,
        });

        return {
          fileUrl: await getSignedUrl(client, command, {
            expiresIn: 3600,
          }),
        };
      },

      async readText({ storageUri }) {
        const { key } = parseAndValidateStorageUri(storageUri);
        const response = await client.send(
          new GetObjectCommand({
            Bucket: bucketName,
            Key: key,
          }),
        );

        return response.Body?.transformToString() ?? null;
      },
    };
  },
});
```

## Helper Utilities

The `@hot-updater/plugin-core` package provides storage helpers.

### `createStorageKeyBuilder`

Builds storage keys with an optional base path and a filename.

```typescript
const getStorageKey = createStorageKeyBuilder("releases");

getStorageKey("bundle-id", "bundle.zip");
// "releases/bundle-id/bundle.zip"
```

### `getStorageUploadFilePath`

Returns a local file path from a file upload source and throws if the runtime
received bytes instead.

```typescript
const filePath = getStorageUploadFilePath(source);
```

### `parseStorageUri`

Parses storage URIs into bucket and key components.

```typescript
const { bucket, key } = parseStorageUri(
  "custom-s3://my-bucket/releases/bundle.zip",
  "custom-s3",
);
// bucket: "my-bucket"
// key: "releases/bundle.zip"
```

### `getContentType`

Returns a MIME type based on a file extension.

```typescript
const contentType = getContentType("bundle.js");
// "application/javascript"
```

## CLI Configuration

Use your custom plugin in `hot-updater.config.ts`.

```typescript
import { defineConfig } from "@hot-updater/core";

import { customStorage } from "./customStorage";

export default defineConfig({
  storage: customStorage({
    region: "us-east-1",
    credentials: {
      accessKeyId: process.env.AWS_ACCESS_KEY_ID!,
      secretAccessKey: process.env.AWS_SECRET_ACCESS_KEY!,
    },
    bucketName: process.env.BUCKET_NAME!,
    basePath: "releases",
  }),
});
```

The configured plugin must implement `upload`, `downloadFile`, and `delete`
for workflows that upload, copy, diff, or remove local files.

## Custom Server Usage

Use the same storage factory with `createHotUpdater` from
`@hot-updater/server`.

```typescript
import { createHotUpdater } from "@hot-updater/server";

import { customStorage } from "./customStorage";

const hotUpdater = createHotUpdater({
  storages: [
    customStorage({
      region: "us-east-1",
      credentials: {
        accessKeyId: process.env.AWS_ACCESS_KEY_ID!,
        secretAccessKey: process.env.AWS_SECRET_ACCESS_KEY!,
      },
      bucketName: process.env.BUCKET_NAME!,
      basePath: "releases",
    }),
  ],
});
```

For Workers, Pages, or other edge runtimes, inject platform bindings into the
storage config directly:

```typescript
const hotUpdater = createHotUpdater({
  storages: [
    myEdgeStorage({
      bucket: env.BUCKET,
      publicBaseUrl: new URL(request.url).origin,
    }),
  ],
});
```

## Storage URI Format

Use a stable URI format:

```typescript
`${supportedProtocol}://${bucketName}/${storageKey}`;
```

The protocol must match `supportedProtocol`. Always validate the bucket or
namespace before reading, writing, or deleting objects.

Avoid storing public URLs as `storageUri` values for provider-backed objects.
The database should keep provider URIs, while `getDownloadUrl` should decide
how to expose a temporary or public HTTP(S) URL to clients.

## Testing

At minimum, cover these cases with unit tests:

- `upload` returns a `storageUri` using the configured protocol and bucket.
- `downloadFile` creates the destination file.
- `delete` rejects storage URIs from a different bucket or namespace.
- `getDownloadUrl` returns an HTTP(S) URL.
- `readText` returns text for an existing metadata object and `null` for a
  missing one.
- Runtime-specific configs pass platform bindings directly instead of relying
  on request context.

## Security

- Never hardcode credentials in plugin code.
- Use environment variables or platform secret stores for sensitive values.
- Validate `storageUri` before reading, downloading, or deleting.
- Return only HTTP(S) URLs from `getDownloadUrl`.
- Keep `readText` on direct provider reads for metadata.
- Prefer short-lived signed URLs when storage objects are private.
