## Prerequisites

Before you begin, make sure the following are ready:

- **Node.js**: Version 20 or later is recommended.
- **AWS Account**: Sign up at [AWS](https://aws.amazon.com) if you don't have one.
- **AWS CLI**: Install the AWS CLI and sign in or configure a profile.

## Required AWS Permissions

Hot Updater requires specific AWS IAM permissions for setup and ongoing usage:

### 1. Initialization (One-Time Setup)

Used for `hot-updater init`:

- `AmazonS3FullAccess`: Create and read S3 buckets.
- `AWSLambda_FullAccess`: Create and update Lambda functions.
- `CloudFrontFullAccess`: Manage CloudFront distributions.
- `IAMFullAccess`: Create IAM roles for Lambda@Edge.
- `SSMFullAccess`: Create Access to SSM Parameters for storing CloudFront key pairs.

### 2. Ongoing Usage

Used for `hot-updater deploy` and `hot-updater console`:

- `AmazonS3FullAccess`: Manage bundles and metadata in the S3 bucket.
- `cloudfront:CreateInvalidation`: Invalidate CloudFront cache after deployments.

For ongoing usage, create a separate access token with limited permissions. Here's a minimal policy example for CloudFront invalidation:

```json
{
  "Effect": "Allow",
  "Action": ["cloudfront:CreateInvalidation"],
  "Resource": "arn:aws:cloudfront::<ACCOUNT_ID>:distribution/<DISTRIBUTION_ID>"
}
```

If this permission is missing, you may encounter the following error:

```
AccessDenied: User: arn:aws:sts::***:assumed-role/***/*** is not authorized to perform: cloudfront:CreateInvalidation on resource: arn:aws:cloudfront::***:distribution/*** because no identity-based policy allows the cloudfront:CreateInvalidation action
```

## Step 1: Install Required Packages

Run the following command to install dependencies:

```package-install
npm install hot-updater --save-dev
```

## Step 2: Configure AWS

Run the initialization script to start the interactive setup process. Use the following command with your preferred package manager:

```package-install
npx hot-updater init
```

2. **Select a Provider**: Select AWS + Lambda@Edge as the provider for handling updates.
   During the setup, you will be prompted to:
3. **Choose AWS Login Method**: Select between current AWS CLI/default session, shared AWS profile, AWS Access Key, or SSO login

6. **Select S3 Bucket**: Choose an existing bucket or create a new one
7. **Create IAM Role**: Create or select an IAM role for Lambda@Edge

9. **Create/Update CloudFront Distribution**: Create or update a CloudFront distribution

Once the setup is complete, `.env.hotupdater` will be generated with your bucket,
region, CloudFront distribution ID, and any profile or key settings needed for the
mode you selected.

```
HOT_UPDATER_S3_BUCKET_NAME=your-s3-bucket-name
HOT_UPDATER_S3_REGION=your-region
HOT_UPDATER_CLOUDFRONT_DISTRIBUTION_ID=your-distribution-id
```

If you choose a shared AWS profile, the generated env file will also include:

```bash
HOT_UPDATER_AWS_PROFILE=your-profile-name
```

If you choose the current AWS CLI/default session mode, no extra credential env
variables are written. The generated config uses the AWS SDK default provider
chain directly.

If you choose access keys, the generated env file will instead include:

```bash
# The generated key may be over-privileged or temporary depending on how you authenticated during init.
HOT_UPDATER_S3_ACCESS_KEY_ID=your-access-key-id
HOT_UPDATER_S3_SECRET_ACCESS_KEY=your-secret-access-key
```

<Callout type="warn">

If you're **not** using the `react-native-dotenv` solution, the tokens from your `.env` file will not be included in your app bundle and are therefore not exposed to risks. However, if you're still concerned,

please refer to the article below for more details:
[Security](/docs/policy/security#plugin-token-protection)

</Callout>

## Step 3: Generated Configurations

During the initialization process, the following file is automatically generated:

- `hot-updater.config.ts`: This file contains the configuration settings for integrating AWS with your project.

If you choose **current AWS CLI/default session**, the generated config uses the
AWS SDK default provider chain instead of embedding access keys:

```ts title="hot-updater.config.ts"
import { bare } from "@hot-updater/bare";
import { fromNodeProviderChain } from "@aws-sdk/credential-providers";
import { s3Storage, s3Database } from "@hot-updater/aws";
import { defineConfig } from "hot-updater";
import { config } from "dotenv";

config({ path: ".env.hotupdater" });

const commonOptions = {
  bucketName: process.env.HOT_UPDATER_S3_BUCKET_NAME!,
  region: process.env.HOT_UPDATER_S3_REGION!,
  credentials: fromNodeProviderChain(),
};

export default defineConfig({
  build: bare({ enableHermes: true }),
  storage: s3Storage(commonOptions),
  database: s3Database({
    ...commonOptions,
    cloudfrontDistributionId:
      process.env.HOT_UPDATER_CLOUDFRONT_DISTRIBUTION_ID!,
  }),
});
```

If you choose **shared AWS profile**, `fromNodeProviderChain()` is replaced with
`fromIni({ profile: process.env.HOT_UPDATER_AWS_PROFILE! })`. If you choose
access keys or SSO, the generated credential block matches that mode instead.

## Step 4: Change `.env` file (Optional)

By this point, the .env file should already be created. The token inside was generated when you ran the `hot-updater init` command.

This step only applies if you selected **access key** or **SSO** during `init`.
If you selected **current AWS CLI/default session** or **shared AWS profile**,
you can keep using the generated credential-provider setup and skip this section.

The token may have excessive permissions due to infrastructure setup or expire if SSO was used.

To avoid issues, update it with a permanent token with `S3FullAccess` and `CloudFrontFullAccess`.

```bash title=".env.hotupdater"
HOT_UPDATER_S3_ACCESS_KEY_ID=your-access-key-id
HOT_UPDATER_S3_SECRET_ACCESS_KEY=your-secret-access-key
```

## Step 5: Add HotUpdater to Your Project

The HotUpdater component wraps your application, enabling seamless delivery of updates and fallback UI during updates. Follow these steps to integrate it into your App.tsx:

```tsx title="App.tsx"
import { HotUpdater } from "@hot-updater/react-native";

function App() {
  return (
    <View>
      <Text>Hello World</Text>
    </View>
  );
}

export default HotUpdater.wrap({
  baseURL: "https://<your-cloudfront-distribution-url>/api/check-update", // [!code ++]
  updateStrategy: "appVersion", // or "fingerprint" // [!code ++]
  updateMode: "auto",
  requestHeaders: {
    // if you want to use the request headers, you can add them here
  },
  fallbackComponent: ({ progress, status }) => (
    <View
      style={{
        flex: 1,
        padding: 20,
        borderRadius: 10,
        justifyContent: "center",
        alignItems: "center",
        backgroundColor: "rgba(0, 0, 0, 0.5)",
      }}
    >
      {/* You can put a splash image here. */}

      <Text style={{ color: "white", fontSize: 20, fontWeight: "bold" }}>
        {status === "UPDATING" ? "Updating..." : "Checking for Update..."}
      </Text>
      {progress > 0 ? (
        <Text style={{ color: "white", fontSize: 20, fontWeight: "bold" }}>
          {Math.round(progress * 100)}%
        </Text>
      ) : null}
    </View>
  ),
})(App);
```

<Callout type="info">
The managed AWS runtime exposes the public update-check endpoints and `/version`. Internally, it is equivalent to `createHotUpdater({ routes: { updateCheck: true, version: true, bundles: false } })`, so bundle management endpoints are not mounted on the public function URL.
</Callout>
