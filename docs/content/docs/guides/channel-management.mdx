## Overview

Channels work for both environment management and multi-app scenarios. You can use them to separate environments like dev, staging, and production, or to manage completely different apps (e.g., app2, app3, app4) with their own channels.

- **Default Channel**: Apps use the `production` channel when no channel is specified.
- **Environment Management**: Separate your environments (`dev`, `staging`, `production`) to control which updates go where.

## Setting Up Channels

### 1. Specify Channel in Configuration File

    ```package-install

      <resources>
          <string name="app_name">HotUpdaterExample</string>
           // [!code ++]
          <string name="hot_updater_channel" moduleConfig="true">your_set_channel</string>
      </resources>
      ```
      </Tab>
    </Tabs>
  </Tab>
  <Tab value="Expo">
    For Expo, you need to add the channel to the plugin configuration in your `app.json` file.

    ```json title="app.json"
    {
      "expo": {
        "plugins": [
          [
            "@hot-updater/react-native",
            {
              "channel": "<your-channel-name>"
            }
          ]
        ]
      }
    }
    ```
    Replace `<your-channel-name>` with your desired channel (e.g., `production`).

    After modifying `app.json`, run `prebuild` to apply the changes to the native projects.

    ```package-install
    npx expo prebuild --clean
    ```

## Custom Configuration Paths

For projects with non-standard structures or iOS app extensions, you can configure custom paths for platform-specific configuration files in your `hot-updater.config.ts`:

```typescript title="hot-updater.config.ts"
import { defineConfig } from "hot-updater";

export default defineConfig({
  // ... other configurations
  platform: {
    ios: {
      infoPlistPaths: [
        "ios/YourApp/Info.plist",
        "ios/YourAppExtension/Info.plist", // Include extension Info.plist
        "ios/NotificationService/Info.plist"
      ]
    },
    android: {
      stringResourcePaths: [
        "android/app/src/main/res/values/strings.xml",
        "android/app/src/debug/res/values/strings.xml" // Include additional resource files
      ]
    }
  }
});
```

This is useful when:
- Your iOS project has app extensions (widgets, share extensions, notification services)
- You have multiple Android build variants with different resource paths
- Your project doesn't follow the standard React Native file structure

**Important**: Configure these custom paths before running `hot-updater channel set` commands to ensure all relevant files are updated with the new channel.

### 3. Specify Channel Deployment

You can also specify the target channel directly during deployment:

```package-install
npx hot-updater deploy -p <"ios" | "android"> -c "<channel>"
```

Replace `<channel>` with your target channel name (e.g., `dev`, `staging`, `production`).

## Retrieving the Current Channel

Use the `HotUpdater.getChannel()` function to retrieve the current release channel within your app.
We recommend displaying this value in your app UI to visually confirm that it's using the intended update channel.

### Example Usage

```tsx
import { HotUpdater } from "@hot-updater/react-native";
import { Text, View } from "react-native";

function App() {
  const channel = HotUpdater.getChannel();

  return (
    <View>
      <Text>Current Channel: {channel}</Text>
    </View>
  );
}

export default HotUpdater.wrap({
  baseURL: "<your-update-server-url>",
  updateStrategy: "appVersion", // or "fingerprint"
  updateMode: "auto",
})(App);
```

## Switching Channels at Runtime

In addition to setting the channel at build time, you can dynamically switch channels at runtime using `HotUpdater.getBundleChannel()`. This allows users or developers to change which channel receives updates without rebuilding the app.

### When to Switch Channels

- **User opt-in**: Allow users to choose between production, beta, or development channels
- **Testing**: Switch between channels during development without rebuilding
- **Staged rollouts**: Gradually roll out updates by routing users to different channels
- **A/B testing**: Test different update versions with different user groups

### Example: Channel Switcher

```tsx
import { HotUpdater } from "@hot-updater/react-native";
import { TouchableOpacity, Text } from "react-native";

async function switchChannel(channelName: string) {
  const bundleInfo = await HotUpdater.getBundleChannel(channelName, {
    updateStrategy: "appVersion",
  });

  if (!bundleInfo) {
    console.log(`No bundle available for: ${channelName}`);
    return;
  }

  await bundleInfo.updateBundle();
  await HotUpdater.reload();
}

export default function Settings() {
  return (
    <>
      <TouchableOpacity onPress={() => switchChannel("production")}>
        <Text>Switch to Production</Text>
      </TouchableOpacity>
      <TouchableOpacity onPress={() => switchChannel("staging")}>
        <Text>Switch to Staging</Text>
      </TouchableOpacity>
    </>
  );
}
```

### Important Notes on Runtime Switching

- **Downloads update**: Unlike a simple channel preference change, this actually downloads and installs the bundle from the new channel
- **Requires app reload**: Call `HotUpdater.reload()` to restart the app with the new bundle
- **Checks availability**: Returns `null` if no bundle is available for the requested channel
- **Requires server communication**: Must query the update server for the target channel's bundle

## Resetting to Original Bundle

Use `HotUpdater.resetToOriginalBundle()` to completely reset the app to its original state, clearing all downloaded bundles and restoring the original channel.

### When to Reset

- **Troubleshooting**: Verify that issues are related to updates, not the original app
- **User opt-out**: Allow users to uninstall OTA updates and return to the original app
- **Clean state**: Reset to a known-good state during testing or debugging
- **Error recovery**: Recover from a problematic downloaded bundle

### Example: Reset Button in Settings

```tsx
import { HotUpdater } from "@hot-updater/react-native";
import { TouchableOpacity, Text, View, Alert } from "react-native";

function SettingsScreen() {
  const handleReset = async () => {
    Alert.alert(
      "Reset to Original Bundle",
      "This will remove all downloaded updates and restore the original app.",
      [
        { text: "Cancel", onPress: () => {}, style: "cancel" },
        {
          text: "Reset",
          onPress: async () => {
            try {
              const success = await HotUpdater.resetToOriginalBundle();
              if (success) {
                Alert.alert("Success", "Reset complete. Restarting app...");
                // Reload the app with the original bundle
                await HotUpdater.reload();
              }
            } catch (error) {
              Alert.alert("Error", `Failed to reset: ${error}`);
            }
          },
          style: "destructive",
        },
      ]
    );
  };

  return (
    <View>
      <TouchableOpacity onPress={handleReset}>
        <Text style={{ color: "red" }}>Reset to Original Bundle</Text>
      </TouchableOpacity>
    </View>
  );
}

export default SettingsScreen;
```

### What Reset Does

- **Removes all downloaded bundles**: Clears all OTA-installed bundles from storage
- **Clears crash history**: Resets the record of bundles that have crashed
- **Restores original channel**: Reverts to the default channel specified at build time

## Channel Behavior

- Defaults to `production` if no channel is specified.
- Channels ensure separation of different environments (e.g., `dev`, `staging`, `production`) to apply updates accurately.
- Updates can subsequently be deployed using `hot-updater deploy -c <channel>`.
- **Important**: Changing the channel requires rebuilding the native app; simply altering the configuration file or deployment commands will not affect an already built app.
