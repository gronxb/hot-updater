## Overview

Hot Updater uses cohorts to decide whether a device is eligible for a bundle.

- Every install gets a persisted numeric cohort from `1` to `1000`.
- `npx hot-updater deploy -r <percentage>` maps `1..100` to those numeric
  cohorts.
- Omitting `-r` uses `100`, which is a full rollout.
- `-r 0` is also valid when you want to deploy a bundle without exposing it to
  numeric rollout yet.
- Custom cohorts such as `qa`, `dogfood`, or `vip-testers` can be assigned
  directly on the device.

<Callout type="info">
  Percentage rollout applies only to numeric cohorts. Custom cohorts do not
  participate in partial numeric rollout unless you add them to `Target
  Cohorts`.
</Callout>

## Deploy a Bundle with RollOut

Use `-r` during deploy:

```package-install
npx hot-updater deploy -p ios -r 10
npx hot-updater deploy -p ios -r 25
npx hot-updater deploy -p ios -r 100
```

- `-r 10` targets 100 of 1000 numeric cohorts.
- `-r 25` targets 250 of 1000 numeric cohorts.
- `-r 100`, or omitting `-r`, targets the full audience.

Hot Updater keeps the selected numeric cohort set stable per bundle. That means
increasing a bundle from `10%` to `25%` expands the same bundle's eligible set
instead of reshuffling everyone.

## Scenario 1: Numeric Rollout Flow

This scenario covers the normal numeric rollout flow.

In this scenario, assume `HotUpdater.getCohort()` returns `411`.

### See the Current Cohort on Device

Check the current device cohort with `HotUpdater.getCohort()`:

```tsx
import { HotUpdater } from "@hot-updater/react-native";
import { Text } from "react-native";

export function CohortStatus() {
  return <Text>HotUpdater.getCohort(): {HotUpdater.getCohort()}</Text>;
}
```

This is the device state before rollout:

<img src={"/docs/rollout-cohorts/get-cohort-ios.png"} width="320" />

In this example, `HotUpdater.getCohort()` returns `411`, so this device belongs
to cohort `411`.

### Preview the Selected Numeric Cohorts

Assume you deploy the bundle with a `50%` rollout:

```package-install
npx hot-updater deploy -p ios -r 50
```

Open the bundle in the console and click `Preview Cohorts`:

<img src={"/docs/rollout-cohorts/preview_411.png"} />

This opens the numeric cohort preview for that bundle:

<img src={"/docs/rollout-cohorts/preview_cohort.png"} />

If rollout is `50.0%`, Hot Updater selects `500` numeric cohorts out of
`1..1000`. That selection is based on the bundle's `bundleId`, so the result is
stable for that bundle.

<Callout type="info">
  `50%` rollout does not mean cohorts `1..500`. Hot Updater uses a stable
  shuffled set per bundle, and `Preview Cohorts` shows the exact numeric cohorts
  selected for that rollout.
</Callout>

You can read the result like this:

| Check                        | Example                           | Result                                                 |
| ---------------------------- | --------------------------------- | ------------------------------------------------------ |
| Device cohort                | `HotUpdater.getCohort() = 411`    | This device belongs to cohort `411`.                   |
| Rollout percentage           | `50.0%`                           | The bundle targets `500` of `1000` numeric cohorts.    |
| Cohort excluded from preview | `410` is not in the selected list | A device in cohort `410` does not receive this bundle. |
| Cohort included in preview   | `411` is in the selected list     | A device in cohort `411` does receive this bundle.     |

Here is the direct check for cohort `411`:

<img src={"/docs/rollout-cohorts/preview_411.png"} />

In this example, cohort `411` is included in the selected rollout set. Since
the iOS example app above reported `HotUpdater.getCohort(): 411`, this device
is eligible for the rollout and will receive the bundle.

After the app checks for updates and applies the eligible bundle, you end up
with the post-update state below:

<img src={"/docs/rollout-cohorts/final-current-state.png"} width="320" />

This walkthrough shows the normal numeric rollout flow:

1. The device starts with cohort `411`.
2. You deploy with `npx hot-updater deploy -p ios -r 50`.
3. The preview confirms that cohort `411` is included.
4. The device is eligible, so it receives the bundle.

This is why the preview is useful: percentage rollout is not a simple `1..500`
range. Hot Updater keeps a stable shuffled set per bundle, and `Preview
Cohorts` shows the exact numeric cohorts included in that rollout.

## Scenario 2: Manual Cohort Override and Target Cohorts

This scenario builds on top of the existing numeric cohort system. Use it when
you want to manually override the device cohort and explicitly target that
value from the console.

This walkthrough uses `qa` as the example value because that is what the app
screen and console screenshot show.

The same mechanism is not limited to custom names. It can also use the existing
numeric cohort values from the original rollout system.

This is the device state after overriding the cohort to `qa`:

<img src={"/docs/rollout-cohorts/set_cohort.png"} width="320" />

In this example, the app UI already shows that `HotUpdater.getCohort()` is set
to `qa`.

### Override the Device Cohort

Use `HotUpdater.setCohort()` when you want to override the device cohort:

<Tabs items={["Custom name", "Numeric value"]}>
  <Tab value="Custom name">

```tsx
import { HotUpdater } from "@hot-updater/react-native";

function assignQaCohort() {
  HotUpdater.setCohort("qa");
  console.log("Active cohort:", HotUpdater.getCohort());
}
```

  </Tab>
  <Tab value="Numeric value">

```tsx
import { HotUpdater } from "@hot-updater/react-native";

function assignNumericCohort(numericCohort: string) {
  HotUpdater.setCohort(numericCohort);
  console.log("Active cohort:", HotUpdater.getCohort());
}
```

  </Tab>
</Tabs>

This works for both custom cohort names like `qa` and numeric cohort values
from the existing rollout system.

- `HotUpdater.setCohort("qa")` moves the device into a named tester group.
- `HotUpdater.setCohort(numericCohort)` makes the device behave as that numeric
  cohort.
- Numeric override is useful when a bundle has already been deployed to a known
  numeric cohort and you want a tester device to follow that same rollout path.

<Callout type="warn">
  If you need to restore the original value later, call `HotUpdater.getCohort()`
  first and store it before overriding.
</Callout>

### Target Specific Cohorts from the Console

If you want to ship a bundle to a known set of devices, deploy the bundle first,
then edit it in the console:

```package-install
npx hot-updater deploy -p ios -r 0
npx hot-updater console
```

Open the deployed bundle in the console and add `qa` to `Target Cohorts`:

<img src={"/docs/rollout-cohorts/target_cohorts.png"} />

`Target Cohorts` accepts both numeric cohort values and custom cohorts like
`qa`. In this scenario, the bundle uses
`Rollout Percentage = 0.0%` and `Target Cohorts = qa`.

This screenshot uses `qa`, but the same flow also works with numeric target
cohorts from the existing rollout system.

<Callout type="warn">
  `Target Cohorts` overrides percentage rollout. When at least one target cohort
  is set, only matching cohorts receive the update, even if the rollout slider
  is `0%` or would normally exclude that device.
</Callout>

You can read the result like this:

| Check                        | Example                                                               | Result                                                                                     |
| ---------------------------- | --------------------------------------------------------------------- | ------------------------------------------------------------------------------------------ |
| Device cohort override       | `HotUpdater.setCohort("qa")` or `HotUpdater.setCohort(numericCohort)` | The device can be overridden to either a custom cohort or an existing numeric cohort.      |
| Rollout percentage           | `0.0%`                                                                | No numeric cohort receives the bundle through percentage rollout.                          |
| Allowed target cohort values | existing numeric cohort values, `qa`                                  | `Target Cohorts` supports both the original numeric cohort system and custom cohort names. |
| Target cohorts               | `qa` or a numeric cohort value                                        | You can explicitly include either custom or numeric cohorts.                               |
| Non-matching device          | `HotUpdater.getCohort() = dogfood`                                    | This device does not receive the bundle.                                                   |
| Matching device              | `HotUpdater.getCohort() = qa` or the same numeric value               | This device receives the bundle when the same value appears in `Target Cohorts`.           |

This scenario is useful when you want to ship to a known tester group without
opening the bundle to the full numeric rollout audience. It is also useful when
you want to temporarily move a tester device into an already deployed numeric
cohort from the existing rollout system.

### Manual Override QA Flow

1. Read the current cohort on the tester device with `HotUpdater.getCohort()`.
2. Change it to a target value such as `qa` or an existing numeric cohort value with `HotUpdater.setCohort()`.
3. Deploy the bundle with `npx hot-updater deploy -p ios -r 0`.
4. Open the console and set `Target Cohorts` to the same value, such as `qa` or that numeric cohort value.
5. Save the bundle and check for updates on the device.
