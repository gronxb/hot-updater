## Installation

```package-install
npm install @hot-updater/plugin-core --save-dev
```

Install the public conformance suites in the project that tests your plugin:

```package-install
npm install @hot-updater/test-utils vitest --save-dev
```

## Overview

A custom database plugin maps your database driver to Hot Updater's row-level
CRUD contract. The models are fixed:

- `bundles` stores one row per OTA bundle.
- `bundle_patches` stores patch artifacts owned by bundles.
- `bundle_events` stores append-only app-ready Analytics reports.

Hot Updater composes these rows into complete bundles. Your plugin does not
implement bundle staging, patch orchestration, or Console routes.

The configured value is a plugin object:

```ts
export default defineConfig({
  database: myDatabase({ db }),
});
```

## Operation matrix

Implement only the supported model and method combinations:

| Model | `create` | `update` | `delete` | `count` | `findOne` | `findMany` |
| --- | --- | --- | --- | --- | --- | --- |
| `bundles` | Yes | Yes | Yes | Yes | Yes | Yes |
| `bundle_patches` | Yes | No | Yes | Yes | Yes | Yes |
| `bundle_events` | Yes | No | No | Yes | Yes | Yes |

`bundle_events` is append-only. Hot Updater never updates or deletes an event
through the public plugin contract.

Import the canonical row and operation types instead of declaring a second
schema mapping:

```ts
import type {
  BundleEventRow,
  BundlePatchRow,
  BundleRow,
  DatabasePluginImplementation,
} from "@hot-updater/plugin-core";
```

Physical row fields use snake case. Keep those field names in your database
mapping so filtering, sorting, projection, and distinct queries remain
portable across providers.

## Create a plugin

Export a function that accepts provider configuration, then close over the
configured client inside `createDatabasePlugin`:

```ts title="myDatabase.ts"
import {
  createDatabasePlugin,
  type DatabasePluginImplementation,
} from "@hot-updater/plugin-core";

interface MyDatabaseConfig {
  readonly db: MyDatabaseClient;
}

const createImplementation = (
  db: MyDatabaseClient,
): DatabasePluginImplementation => ({
  create: async ({ model, data, select }) => {
    return db.insert({ model, data, select });
  },
  update: async ({ model, where, update, select }) => {
    return db.update({ model, where, update, select });
  },
  delete: async ({ model, where }) => {
    await db.delete({ model, where });
  },
  count: async ({ model, where, distinct }) => {
    return db.count({ model, where, distinct });
  },
  findOne: async ({ model, where, select }) => {
    return db.findOne({ model, where, select });
  },
  findMany: async ({
    model,
    where,
    limit,
    offset,
    orderBy,
    sortBy,
    distinctOn,
    select,
  }) => {
    return db.findMany({
      model,
      where,
      limit,
      offset,
      orderBy,
      sortBy,
      distinctOn,
      select,
    });
  },
});

export const myDatabase = ({ db }: MyDatabaseConfig) =>
  createDatabasePlugin({
    name: "my-database",
    plugin: () => createImplementation(db),
  });
```

`MyDatabaseClient` represents your ORM or database driver. Adapt each method to
the driver's query API. The `DatabasePluginImplementation` annotation keeps
the fixed model and operation inputs typed while you build the mapping.

`createDatabasePlugin` validates inputs and provider results. It also exposes
Analytics support because a row-oriented plugin can append `bundle_events`
safely.

## Implement each operation

### Create

`create` receives a complete row and returns the created row. When `select` is
present, return at least the requested fields.

Reject duplicate IDs in the database. Keep event creation concurrency-safe so
two app-ready reports cannot overwrite each other.

### Update

Only `bundles` can be updated. Hot Updater uses an exact `id` predicate and
expects the updated row, or `null` when the bundle does not exist.

The `update` object never changes the bundle ID.

### Delete

`delete` removes every row matched by its non-empty predicate. It applies to
`bundles` and `bundle_patches` only.

Deleting a bundle must also remove patch rows that reference it. Do not delete
`bundle_events`; historical reports may reference old or unknown bundle IDs.

### Count and reads

`count` returns a non-negative safe integer. It may receive `distinct` fields
for Analytics queries.

`findOne` returns one matching row or `null`. `findMany` returns an array and
receives normalized `limit` and `offset` values. Filtering and ordering happen
before pagination.

Apply `select` at the database boundary when your driver supports projection.
Every requested field must be present in the returned row.

## Filtering and ordering

Each `where` item contains a field, value, and optional operator, connector, or
string comparison mode:

```ts
const rows = await plugin.findMany({
  model: "bundles",
  where: [
    { field: "channel", value: "production" },
    {
      field: "platform",
      operator: "in",
      value: ["ios", "android"],
      connector: "AND",
    },
  ],
  orderBy: [
    { field: "id", direction: "desc" },
  ],
  limit: 50,
  offset: 0,
});
```

Supported operators are `eq`, `ne`, `lt`, `lte`, `gt`, `gte`, `in`,
`not_in`, `contains`, `starts_with`, and `ends_with`, subject to the selected
field's type.

- Missing or empty read predicates match all rows.
- Mutation predicates must not be empty.
- `in: []` matches no rows; `not_in: []` matches all rows.
- String comparisons may use `mode: "sensitive"` or `"insensitive"`.
- Later predicates default to the `AND` connector.
- `orderBy` preserves clause order and may specify `nulls: "first"` or
  `"last"`.
- `distinctOn` selects the first row for each distinct field tuple after
  ordering.

Translate these values to parameterized ORM or database-driver queries. Never
concatenate field values into raw SQL.

## Optional capabilities

### Channel aggregation

Implement `getChannels` when the database can return sorted, distinct bundle
channels efficiently. If omitted, Hot Updater derives channels from bundle
pages.

### Update-check fast path

Implement `getUpdateInfo` when the backend can answer update checks more
efficiently than the shared row-query path.

The optimized result must match the shared behavior for channel defaults,
rollout cohorts, rollback constraints, patches, and manifest fields. Leave the
method out when the database cannot preserve those semantics.

### Transactions

Expose `transaction` only when the database can roll back the complete
callback:

```ts
transaction: async (callback) => {
  return db.transaction(async (transactionDb) => {
    return callback(createImplementation(transactionDb));
  });
},
```

If `transaction` is absent, Hot Updater performs the same writes sequentially.
That fallback is not atomic and may leave earlier writes persisted when a later
write fails.

### Lifecycle

Use `onUnmount` to close resources owned by the plugin implementation. Attach
`onDatabaseUpdated` to the returned plugin only when an aggregate mutation
must invalidate an external cache.

Database plugin operations do not receive request context. Close over the
configured client or binding. Request context remains available to server
handlers and storage plugins.

## Test the plugin

Use both public conformance suites:

```ts title="myDatabase.spec.ts"
import { createDatabaseClient } from "@hot-updater/plugin-core";
import {
  setupDatabaseClientTestSuite,
  setupDatabasePluginTestSuite,
} from "@hot-updater/test-utils";

const lifecycle = {
  name: "my database",
  createPlugin: () => myDatabase({ db }),
  migrate: () => migrateDatabase(db),
  reset: () => resetDatabase(db),
  dispose: () => closeDatabase(db),
};

setupDatabasePluginTestSuite(lifecycle);

setupDatabaseClientTestSuite({
  ...lifecycle,
  createClient: createDatabaseClient,
});
```

The plugin suite covers CRUD inputs, projections, predicates, ordering,
pagination, distinct queries, relations, and optional capabilities. The client
suite verifies bundle aggregation, patch replacement, channels, and bundle
management behavior.

Run these suites against the real database used by the plugin. They should own
their schema, reset data between tests, and close connections after the suite.

## Configure Hot Updater

Use the same plugin factory in the CLI and self-hosted server:

```ts title="hot-updater.config.ts"
import { defineConfig } from "hot-updater";
import { myDatabase } from "./myDatabase";

export default defineConfig({
  database: myDatabase({ db }),
  // build: ...
  // storage: ...
});
```

```ts title="server.ts"
import { createHotUpdater } from "@hot-updater/server";
import { myDatabase } from "./myDatabase";

export const hotUpdater = createHotUpdater({
  database: myDatabase({ db }),
  storages,
});
```

Keep the database schema in sync with the installed Hot Updater version. Add
the `bundle_events` model before enabling client Analytics.

## Blob-backed databases

Use `createBlobDatabasePlugin` only when the backend stores and replaces a
shared snapshot. Blob-backed plugins do not expose Analytics because concurrent
event appends can overwrite each other.

See [Analytics](/docs/guides/analytics) for database support, event ingestion,
and deployment order.
