## Overview

The `standaloneRepository` plugin connects your Hot Updater CLI to a self-hosted backend server. This is the **client-side configuration** for connecting to your server.

<Callout type="info">
**Building a Self-Hosted Server?**

See the [Self-Hosting (Custom) Guide](../custom/overview) for complete server implementation instructions using `@hot-updater/server` with database adapters (Drizzle, Prisma, Kysely, MongoDB) and frameworks (Hono, Express, Elysia).
</Callout>

## Installation

```package-install
npm install @hot-updater/standalone --save-dev
```

## Usage

Configure your CLI to connect to your self-hosted server:

```typescript title="hot-updater.config.ts"
import { defineConfig } from "hot-updater";
import { standaloneRepository } from "@hot-updater/standalone";

export default defineConfig({
  build: /* your build plugin */,
  storage: /* your storage plugin */,
  database: standaloneRepository({
    baseUrl: "http://localhost:3000/hot-updater",
  }),
});
```

## Configuration

The `standaloneRepository` plugin accepts the following options:

```typescript
interface StandaloneRepositoryConfig {
  baseUrl: string;                        // Your server URL
  commonHeaders?: Record<string, string>; // Optional headers (e.g., authentication)
  routes?: Routes;                        // Optional custom route configuration
  getUpdateInfo?: boolean;                // Enable the optional server fast path
}
```

`routes` lets you map the client to a different server shape:

```typescript
interface RouteConfig {
  path: string;
  headers?: Record<string, string>;
}

interface Routes {
  database?: () => RouteConfig;
}
```

### Basic Example

```typescript
database: standaloneRepository({
  baseUrl: "http://localhost:3000/hot-updater",
})
```

### With Authentication

```typescript
database: standaloneRepository({
  baseUrl: process.env.HOT_UPDATER_SERVER_URL!,
  commonHeaders: {
    "Authorization": `Bearer ${process.env.API_TOKEN}`
  }
})
```

### With Custom Routes

```typescript
database: standaloneRepository({
  baseUrl: "https://api.example.com",
  routes: {
    database: () => ({
      path: "/v2/hot-updater/database",
      headers: { "X-Custom-Header": "value" },
    }),
  },
})
```

## Server Contract

`standaloneRepository` is an HTTP client. If you are building a custom
backend, your server must implement the fixed-model database adapter protocol
below. The default endpoint is:

```text
POST {baseUrl}/api/database/v2/:model/:operation
```

<Callout type="info">
If you use `@hot-updater/server`, you do **not** need to implement these
endpoints manually. `createHotUpdater()` provides this contract when bundle
routes are explicitly enabled and authorized.
</Callout>

### Operations

The model and operation URL segments are allowlisted. Request bodies use the
same snake-case rows, predicates, projection, sorting, and pagination as the
[custom database adapter](./custom-database) contract.

| Model | Operation | Request body | Response `data` |
|-------|-----------|--------------|-----------------|
| `bundles` | `create` | `{ data, select? }` | one row |
| `bundles` | `update` | `{ where, update, select? }` | one row or `null` |
| `bundles` | `delete` | `{ where }` | `null` |
| `bundles` | `count` | `{ where? }` | non-negative integer |
| `bundles` | `findOne` | `{ where?, select? }` | one row or `null` |
| `bundles` | `findMany` | `{ where?, sortBy?, limit, offset, select? }` | row array |
| `bundle_patches` | `create` | `{ data, select? }` | one row |
| `bundle_patches` | `delete` | `{ where }` | `null` |
| `bundle_patches` | `findMany` | `{ where?, sortBy?, limit, offset, select? }` | row array |
| `channels` | `create` | `{ data, select? }` | one row |
| `channels` | `findOne` | `{ where?, select? }` | one row or `null` |
| `channels` | `findMany` | `{ where?, sortBy?, limit, offset, select? }` | row array |

Every successful response is JSON with a `data` field:

```json
{
  "data": [{ "id": "production" }]
}
```

Protocol errors use an `error` field with a stable code and message:

```json
{
  "error": {
    "code": "unsupported-operation",
    "message": "Unsupported database operation: channels.delete."
  }
}
```

### Optional update-check fast path

Set `getUpdateInfo: true` only when the server adapter implements the optional
capability. The client then sends `GetBundlesArgs` to
`bundles/getUpdateInfo`. A server without that capability returns `501`.

The HTTP protocol does not expose transactions. Multi-row aggregate writes
therefore use the shared client's documented sequential, non-atomic fallback.

### Minimal `@hot-updater/server` Example

If you want this contract without implementing it yourself, use
`@hot-updater/server`:

```typescript
import { createHotUpdater } from "@hot-updater/server";

const hotUpdater = createHotUpdater({
  database,
  storages: [storage],
  basePath: "/hot-updater",
  routes: {
    updateCheck: true,
    bundles: true,
  },
});
```

This mounts the database route consumed by `standaloneRepository` under
`/hot-updater/api/database/v2/*`. `routes.bundles: true` is the explicit
opt-in for this write-capable API; protect it at the framework or reverse
proxy layer, and configure matching `commonHeaders` in
`standaloneRepository`.
