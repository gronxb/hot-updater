## Usage

Use `HotUpdater.setReloadMethod()` to control how [`HotUpdater.reload()`](./reload) is executed.

This API is available on both Android and iOS so app code can stay symmetric.
The default is `processRestart`.
On Android that means a cold app restart.
On iOS the same default behaves like the normal reload flow.

```tsx
import { HotUpdater } from "@hot-updater/react-native";

HotUpdater.setReloadMethod("processRestart");
```

## Why this exists

On Android, the default React Native reload path is an in-process reload.
That means React tears down the current instance and creates a new one inside the same process.

In release apps with native modules, this can be unsafe:

- the old React instance is being destroyed
- a native module or background callback still tries to talk to JS
- the new React instance is not fully ready yet
- the app can soft-crash, blank, or fully crash depending on timing

`processRestart` avoids that class of problem by applying the update through a cold app restart instead of an in-process React reload.

## Supported methods

### `reload`

Uses the normal React Native reload behavior.

- Android: in-process React reload
- iOS: in-process React reload

```tsx
HotUpdater.setReloadMethod("reload");
```

### `processRestart` (default)

Uses Android process restart when available.

- Android: cold process restart
- iOS: behaves like normal `reload`

```tsx
HotUpdater.setReloadMethod("processRestart");
```

This is the default because Android release reloads can crash when native modules touch React while the instance is being recreated.

### `custom`

Runs a custom JS handler on both Android and iOS.

```tsx
HotUpdater.setReloadMethod("custom", async () => {
  await BrownfieldModule.reload();
});
```

This is the recommended choice for brownfield apps where the host app needs to decide how to restore the correct native container or screen.

## API

```ts
HotUpdater.setReloadMethod("reload");
HotUpdater.setReloadMethod("processRestart");
HotUpdater.setReloadMethod("custom", () => void | Promise<void>);
```

## When to use which

- Use `reload` only if you explicitly want the old in-process React Native reload behavior
- Use `processRestart` for the default, safer Android behavior
- Use `custom` when a brownfield host app must coordinate the reload itself

## Interaction with `reloadOnForceUpdate`

When `reloadOnForceUpdate` triggers `HotUpdater.reload()` automatically, it uses the method you configured here.

That means:

- `reload` opts force updates back into the old in-process behavior
- `processRestart` keeps automatic force updates on Android cold restarts by default
- `custom` delegates automatic force updates to your handler
