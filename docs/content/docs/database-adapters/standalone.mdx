## Overview

The `standaloneRepository` compatibility adapter connects your Hot Updater CLI to a self-hosted backend server. This is the **client-side configuration** for connecting to your server.

> `standaloneRepository` is a compatibility bridge over the existing aggregate
> bundle routes, not a strict fixed-model v2 database adapter. The legacy
> channels response contains names only, so the bridge deterministically uses
> each channel name as its low-row `id`; it cannot preserve a separate caller-
> supplied channel ID. Use a custom database adapter when independent channel
> IDs or the public v2 conformance contract are required.

<Callout type="info">
**Building a Self-Hosted Server?**

See the [Self-Hosting (Custom) Guide](../custom/overview) for complete server implementation instructions using `@hot-updater/server` with database adapters (Drizzle, Prisma, Kysely, MongoDB) and frameworks (Hono, Express, Elysia).
</Callout>

## Installation

```package-install
npm install @hot-updater/standalone --save-dev
```

## Usage

Configure your CLI to connect to your self-hosted server:

```typescript title="hot-updater.config.ts"
import { defineConfig } from "hot-updater";
import { standaloneRepository } from "@hot-updater/standalone";

export default defineConfig({
  build: /* your build plugin */,
  storage: /* your storage plugin */,
  database: standaloneRepository({
    baseUrl: "http://localhost:3000/hot-updater",
  }),
});
```

## Configuration

The `standaloneRepository` adapter accepts the following options:

```typescript
interface StandaloneRepositoryConfig {
  baseUrl: string;                        // Your server URL
  commonHeaders?: Record<string, string>; // Optional headers (e.g., authentication)
  supportsBundleEvents?: true;            // Explicit Analytics capability opt-in
  routes?: Routes;                        // Optional custom route configuration
}
```

`routes` lets you map the client to a different server shape:

```typescript
interface RouteConfig {
  path: string;
  headers?: Record<string, string>;
}

interface Routes {
  appendEvent?: () => RouteConfig;
  bundleEventAnalytics?: (bundleId: string) => RouteConfig;
  bundleEventSummary?: (bundleId: string) => RouteConfig;
  bundleEventOverview?: () => RouteConfig;
  create?: () => RouteConfig;
  update?: (bundleId: string) => RouteConfig;
  list?: () => RouteConfig;
  channels?: () => RouteConfig;
  retrieve?: (bundleId: string) => RouteConfig;
  delete?: (bundleId: string) => RouteConfig;
  installationHistory?: (installId: string) => RouteConfig;
  installations?: () => RouteConfig;
}
```

### Basic Example

```typescript
database: standaloneRepository({
  baseUrl: "http://localhost:3000/hot-updater",
})
```

### With Authentication

```typescript
database: standaloneRepository({
  baseUrl: process.env.HOT_UPDATER_SERVER_URL!,
  commonHeaders: {
    "Authorization": `Bearer ${process.env.API_TOKEN}`
  }
})
```

### With Custom Routes

```typescript
database: standaloneRepository({
  baseUrl: "https://api.example.com",
  routes: {
    create: () => ({
      path: "/v1/hot-updater/api/bundles",
      headers: { "X-Custom-Header": "value" }
    }),
    update: (bundleId) => ({
      path: `/v1/hot-updater/api/bundles/${bundleId}`,
      headers: { "X-Custom-Header": "value" }
    }),
    list: () => ({
      path: "/v1/hot-updater/api/bundles"
    }),
    channels: () => ({
      path: "/v1/hot-updater/api/bundles/channels"
    }),
    retrieve: (bundleId) => ({
      path: `/v1/hot-updater/api/bundles/${bundleId}`
    }),
    delete: (bundleId) => ({
      path: `/v1/hot-updater/api/bundles/${bundleId}`
    }),
    appendEvent: () => ({ path: "/v1/hot-updater/events" }),
    bundleEventOverview: () => ({
      path: "/v1/hot-updater/api/installations/overview"
    }),
    installations: () => ({
      path: "/v1/hot-updater/api/installations"
    })
  }
})
```

## Server Contract

`standaloneRepository` is an HTTP client. If you are building a custom
backend, your server must implement the bundle-management contract below.

<Callout type="info">
If you use `@hot-updater/server`, you do **not** need to implement these
endpoints manually. `createHotUpdater()` provides this contract when bundle
routes are explicitly enabled and authorized, including
`GET /api/bundles/channels`.
</Callout>

### Contract Types

```typescript
interface PaginationInfo {
  total: number;
  hasNextPage: boolean;
  hasPreviousPage: boolean;
  currentPage: number;
  totalPages: number;
  nextCursor?: string | null;
  previousCursor?: string | null;
}

interface DataResponse<TData> {
  data: TData;
}

interface Paginated<TData> extends DataResponse<TData> {
  pagination: PaginationInfo;
}

type PaginatedResult = Paginated<Bundle[]>;
type ChannelsResponse = DataResponse<{ channels: string[] }>;
```

### Default Routes

These are the default routes used by `standaloneRepository`. You can remap
them with the `routes` option shown above.

| Method | Default route | Request body | Success response |
|--------|---------------|--------------|------------------|
| POST | `/api/bundles` | `Bundle[]` | `{ "success": true }` |
| PATCH | `/api/bundles/:id` | `Partial<Bundle>` | `{ "success": true }` |
| GET | `/api/bundles` | none | `Paginated<Bundle[]>` |
| GET | `/api/bundles/:id` | none | `Bundle` |
| DELETE | `/api/bundles/:id` | none | `{ "success": true }` |
| GET | `/api/bundles/channels` | none | `ChannelsResponse` |

When `supportsBundleEvents: true`, the following optional routes are also
required. Leaving the flag unset or false is the provider-neutral default and
causes the Console to hide Analytics and installation UI without making event
requests.

| Method | Default route | Purpose |
|--------|---------------|---------|
| POST | `/events` | Append one bundle event |
| GET | `/api/bundles/:id/events/summary` | Distinct installed/recovered totals |
| GET | `/api/bundles/:id/events/analytics` | Windowed series, cohorts, and recent events |
| GET | `/api/installations/overview` | Identity-free latest-installation bundle counts |
| GET | `/api/installations` | Paginated identity search |
| GET | `/api/installations/:installId/events` | Paginated installation history |

Only enable the flag when all routes implement the same append-only semantic
contract as `DatabaseBundleEventService`: event inserts are concurrency-safe,
latest-installation results use the newest event by timestamp and event ID,
historical identity matches return that installation's latest row, and overview
counts include each installation exactly once. Unknown or deleted bundle IDs
must remain in overview results. The server limits public event bodies to 16
KiB and individual string fields to 1024 characters; custom route
implementations should enforce equivalent or stricter limits before parsing or
persisting input.

### Bundle Shape

Create requests send full `Bundle` objects. Update requests send
`Partial<Bundle>` and, if an `id` is present in the body, it must match the
`:id` route parameter.

```typescript
interface Bundle {
  id: string;
  platform: "ios" | "android";
  shouldForceUpdate: boolean;
  enabled: boolean;
  fileHash: string;
  storageUri: string;
  gitCommitHash: string | null;
  message: string | null;
  channel: string;
  targetAppVersion: string | null;
  fingerprintHash: string | null;
  metadata?: {
    app_version?: string;
  };
  manifestStorageUri?: string | null;
  manifestFileHash?: string | null;
  assetBaseStorageUri?: string | null;
  patches?: Array<{
    baseBundleId: string;
    baseFileHash: string;
    patchFileHash: string;
    patchStorageUri: string;
  }> | null;
  patchBaseBundleId?: string | null; // Deprecated; use patches
  patchBaseFileHash?: string | null; // Deprecated; use patches
  patchFileHash?: string | null;     // Deprecated; use patches
  patchStorageUri?: string | null;   // Deprecated; use patches
  rolloutCohortCount?: number | null;
  targetCohorts?: string[] | null;
}
```

### List Bundles Contract

`GET /api/bundles` must accept the pagination and filter query params used by
the console and `standaloneRepository`, and return a paginated JSON body.

Cursor-based pagination is the official contract. `page` is an optional
positive integer used alongside cursors when the caller needs stable page
numbers.

Supported query params:

| Query param | Type | Notes |
|-------------|------|-------|
| `channel` | `string` | Exact match |
| `platform` | `"ios" \| "android"` | Exact match |
| `enabled` | `boolean` | Exact match |
| `idEq` | `string` | Exact bundle ID |
| `idGt`, `idGte` | `string` | Bundle ID lower bound |
| `idLt`, `idLte` | `string` | Bundle ID upper bound |
| `idIn` | repeated `string` | Match any listed bundle ID |
| `targetAppVersion` | `string \| "null"` | Exact app version or null |
| `targetAppVersionNotNull` | `"true"` | Require a non-null app version |
| `targetAppVersionIn` | repeated `string` | Match any listed app version |
| `fingerprintHash` | `string \| "null"` | Exact fingerprint or null |
| `limit` | `number` | Window size |
| `page` | `number` | Optional positive page number for stable page-aligned requests |
| `after` | `string` | Fetch the next window after this bundle ID |
| `before` | `string` | Fetch the previous window before this bundle ID |

Expected response:

```json
{
  "data": [{ "id": "bundle-id", "channel": "production" }],
  "pagination": {
    "total": 1,
    "hasNextPage": false,
    "hasPreviousPage": false,
    "currentPage": 1,
    "totalPages": 1,
    "nextCursor": null,
    "previousCursor": null
  }
}
```

`standaloneRepository` reads pagination metadata from the response body. It
does not derive pagination from headers. `currentPage` and `totalPages` are
kept for backwards compatibility; `nextCursor` and `previousCursor` are the
preferred navigation fields.

If your standalone server is backed by object storage or manifest files, keep
`GET /api/bundles` on an index-backed path once the index is warm. Cursor
pages should not require rescanning all manifests, and this should be covered
by deterministic read-counter regression tests.

### Retrieve And Channels Contract

- `GET /api/bundles/:id` should return `200` with a `Bundle` body or `404` if
  the bundle does not exist.
- `GET /api/bundles/channels` must return:

```json
{
  "data": {
    "channels": ["production", "staging"]
  }
}
```

### Minimal `@hot-updater/server` Example

If you want this contract without implementing it yourself, use
`@hot-updater/server`:

```typescript
import { createHotUpdater } from "@hot-updater/server";

const hotUpdater = createHotUpdater({
  database,
  storages: [storage],
  basePath: "/hot-updater",
  routes: {
    updateCheck: true,
    bundles: true,
  },
});
```

This mounts the bundle routes consumed by `standaloneRepository` under
`/hot-updater/api/bundles*`. `routes.bundles: true` is the explicit opt-in for
those management APIs; protect them at the framework or reverse proxy layer,
and configure matching `commonHeaders` in `standaloneRepository`.
